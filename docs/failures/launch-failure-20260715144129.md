## Failures Encountered

### 1. Codex 启动报错: Error: failed to start proxy: proxy did not start within 5 seconds
**Symptom**: Codex 启动报错: Error: failed to start proxy: proxy did not start within 5 seconds

**Root Cause**: ensure_proxy_running 中执行 cmd.arg("proxy") 但 clap 定义中 Proxy 已变为需要子命令的枚举(ProxyCommand::Start/Stop)，导致 cct proxy 解析失败进程静默退出，Unix socket 从未创建

**Fix Applied**: 在 cmd.arg("proxy") 后加 .arg("start")，即变为 cct proxy start

```diff
- cmd.arg("proxy")
-     .stdin(std::process::Stdio::null())
+ cmd.arg("proxy")
+     .arg("start")
+     .stdin(std::process::Stdio::null())
```

## Patterns Learned

- spawn process exits silently with stderr discarded
- clap subcommand enum change breaks argument passing
- socket polling timeout masks real startup failure

## Summary

0.2.0 版本中，启动 Codex profile（非 subscription 模式）时，ensure_proxy_running 会 spawn cct proxy 作为守护进程。但由于 cct proxy stop 功能引入了 ProxyCommand 枚举，proxy 从独立子命令变为需要子命令的父命令，而 spawn 代码未同步更新，仍然只传 proxy 不传 start。clap 解析失败后进程静默退出（stderr 被丢弃到 null），Unix socket 从未创建，5秒超时后返回错误。修复：加 .arg("start")。未有测试覆盖因为 ensure_proxy_running 依赖实际 spawn 进程和 Unix socket，属于 effectful 边界代码，缺乏 mock/stub 测试。
