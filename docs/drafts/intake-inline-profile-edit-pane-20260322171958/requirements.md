# Requirements: Inline profile edit pane for selected profiles

## 1. Problem Statement

Today `cct` uses two unrelated edit flows: `a` opens an inline five-field form, while `e` suspends the TUI and opens the raw config file in `$EDITOR`. That makes simple profile changes slower than necessary and breaks the in-app workflow. The code already has backend-aware add-form rendering in `AppMode::AddForm`, but there is no equivalent inline edit path for an existing selected profile.

## 2. Desired Outcome

When the user presses `e` on a selected profile, `cct` should enter an inline editor pane that behaves like the existing add form, but starts with the selected profile's current values filled in. The user can update either Claude or Codex profiles, including renaming the profile, and save changes back to the same entry. If the user changes the name, save must fail when another profile already uses that name. Fields not represented in the current form, such as `extra_args` or unedited env entries, must be preserved exactly instead of being dropped.

## 3. Constraints
- **Tech stack**: Rust TUI app using ratatui, crossterm, serde, toml, and `toml_edit`
- **Hard boundaries**:
  - Reuse the existing add-form interaction model and backend-specific field layout rather than inventing a separate editor UI
  - Support both Claude and Codex profile editing in the first version
  - Allow profile rename, but reject duplicate names using the same case-insensitive semantics as existing profile lookup
  - Replace `[e]` as external config editing; do not keep a second raw-editor path in this change
  - Preserve fields not shown in the form, especially `extra_args` and existing env keys that are not directly edited
  - Avoid destructive config rewrites that erase comments or unrelated profile data

## 4. Scope
### In Scope
- Add an inline edit-mode entry path from `[e]` for the currently selected profile
- Prefill the form from the selected profile using backend-aware field mapping for Claude and Codex
- Save edited values back into the existing profile entry rather than appending a new one
- Allow name changes with duplicate-name validation
- Preserve untouched fields and metadata when saving edited profiles
- Update TUI labels, titles, and footer help so users can distinguish add vs edit behavior
- Update tests for app state, persistence logic, and TUI copy affected by the new edit flow

### Out of Scope
- Keeping `[e]` as a shortcut to open the raw config file in an external editor
- Adding new editable fields beyond the existing form surface
- Refactoring the entire form system beyond what is needed to support add and edit modes cleanly
- Bulk profile management such as delete, reorder, or clone

## 5. Acceptance Criteria
- [ ] Pressing `[e]` with a selected Claude or Codex profile opens an inline pane shaped like the add form, prefilled with that profile's current values
- [ ] Saving an edited profile updates the existing profile entry instead of creating a new profile
- [ ] Renaming is allowed, but save is rejected if another profile already has the target name
- [ ] Claude and Codex editing both use their existing backend-specific field semantics
- [ ] Untouched fields not exposed in the form, including `extra_args` and unrelated env entries, remain unchanged after save
- [ ] `[e]` no longer opens the external config editor in the first-version implementation
- [ ] The TUI copy and help text reflect the inline edit behavior clearly enough that add and edit actions are distinguishable
- [ ] Automated tests cover duplicate handling, prefill mapping, update-vs-append behavior, and field preservation
